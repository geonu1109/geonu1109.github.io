똑똑하게 시간 데이터 관리하기

3계층 구조 기반의 예시와 함께하는 시간 정보의 생성, 연산, 저장에 관한 고찰



안녕하세요. 저는 비즈서비스개발팀 전건우입니다. 저는 현재 iOS 앱 개발을 담당하고 있습니다. 클라이언트 개발을 하다보면 서버 개발자와 소통하며 함께 API 명세를 만드는 경우도 있지만, 이미 존재하는 API를 가져다 쓰는 경우도 많습니다. 이 때, 문서화가 잘 되어있지 않고 해당 API를 만든 개발자도 남아있지 않은 경우 궁금증이 생기면 직접 데이터를 일일히 확인해봐야 하기 때문에 필요 이상으로 시간이 소요되곤 합니다. API를 사용할 때 가장 자주 발생한 두 가지 문제 상황은 첫째로 응답의 각 키 이름을 보고 해당 데이터의 의미를 유추할 수 없는 경우이고 둘째로 받은 값이 무엇을 표현하는 지 알 수 없는 경우입니다. 오늘 이야기할 주제는 두번째 데이터의 값을 알 수 없는 경우 중 시간 데이터 표현에 대해 이야기하고자 합니다. 시간 데이터는 여러가지 종류가 있지만 오늘 이야기할 시간은 로그 기록 시점, 게시글 등록 시점 등 시간대에 유일하게 존재하는 특정 시점을 나타내는 시간 데이터에 집중해서 얘기해보겠습니다.

시간 데이터는 최소 세 가지 정보의 조합으로 이루어집니다. 바로 날짜와 시각 그리고 시간대입니다. 이 셋 중 하나만 빠지더라도 시간 데이터가 가르키는 시점을 파악하기 어려워집니다. 시간 데이터를 표현하기 위한 여러가지 방법이 있지만 제가 이번 세미나를 통해 제안하고자 하는 방식은 바로 유닉스타임입니다. 이를 우리 회사에서 많이 사용하고 있는 ISO8601과 비교하면서 어떤 이점이 있는지 알아보겠습니다.

먼저 각 표기법에 대한 간략한 설명입니다. ISO8601은 문자열로 시간 정보를 표현합니다. 시간 정보 표기 양식이 다른 국가간의 오해를 줄이기 위한 목적으로 만들어졌습니다. 왼쪽으로 갈 수록 큰 단위를 나타내며 오른쪽으로 갈 수록 작은 단위를 나타냅니다. 또한 별도로 시간대를 표기하여 어느 시간대에서 측정된 시간인지를 나타냅니다. 유닉스 시간은 유닉스 계열 운영체제에서 시간을 나타내기 위한 목적으로 만들어졌습니다. 1970년 1월 1일 0시 0분 0초를 기준으로 몇 초가 지났는지를 숫자로 나타냅니다. 시간대는 항상 표준시간대로 고정됩니다.

첫번째로는 유닉스시간이 ISO 8601에 비해 연산이 편리합니다. 문자열로 표시된 시간 정보에 1초 후 같은 단순한 연산을 적용하려고 할 때에도 해당 문자열을 파싱하고 자리수 넘김 처리를 해주어야 합니다. 반면 유닉스 타임의 경우 정렬과 같이 복잡한 연산을 하는 경우에도 별도의 전처리 없이 일반적인 숫자를 정렬하듯 정렬할 수 있습니다.

두번째로는 유닉스시간을 사용하면 ISO 8601에 비해 같은 정보를 더 적은 용량으로 기록할 수 있습니다. 유닉스시간으로 시간정보를 기록하는 경우 2038년 이내라면 4바이트로, 8바이트를 사용하면 거의 무제한적으로 시간을 기록할 수 있습니다. 그러나 ISO 8601을 사용하는 경우 최소17바이트(ex 20200717T091023Z)가 필요합니다.

마지막으로 유닉스시간이 ISO 8601에 비해 가독성이 떨어집니다. 문자열이 어떻게 단순 숫자의 나열보다 가독성이 떨어지냐고 의아해하실 분들이 계실것입니다. 그러나 여기서 말씀드리는 가독성을 판단하는 기준은 인간이 아니라 컴퓨터입니다. ISO 8601이 시간 정보 교환을 위한 국제 표준 양식인 것은 맞지만 이는 컴퓨터가 아닌 인간을 위해 지정된 양식입니다. 따라서 이는 사람 사이의 정보 교환을 위한 몇가지 원칙을 정하고 그 원칙에만 부합하면 같은 시간 정보라도 여러가지 방법으로 표현할 수 있습니다. 시간 정보를 다루기 위해 반드시 파싱작업이 필요한 것을 생각하면 큰 부담으로 작용합니다. 파서 자체도 비대해져야하며 파서가 해석할 수 없는 방식으로 문자열이 전달될 수도 있습니다. 그러나 유닉스시간으로 기록된 시간은 항상 일정한 방식으로 전달됩니다.

정리하겠습니다. 유닉스시간은 ISO 8601에 비해 속도도 빠르고, 용량도 적게 차지하며, 오류 발생 가능성이 낮습니다. 이는 ISO 8601은 태생이 사람과 사람 사이의 시간 정보 교환을 염두해두고 만들어진 것이기 때문입니다. 따라서 소프트웨어에서는 시간정보를 기록하기 위해 유닉스타임을 사용하는 것이 바람직합니다.

대부분의 DBMS나 앱 개발을 위한 프레임워크는 시간 데이터를 표현하기 위한 자체 자료형을 가지고 있습니다. 따라서 개발자들은 큰 고민 없이도 시간 정보를 정말 잘 표현하고 다룰 수 있습니다. 그러나 시간 정보를 표현하기 위한 자체 자료형을 가지지 않은 DBMS나 앱 프레임워크를 다루거나, 네트워크 통신이나 DB 쿼리를 최적화하기 위해서는 이러한 고민을 한번쯤 해보는 것이 좋지 않을까 생각해봅니다.
